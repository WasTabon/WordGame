# WordGame — Iteration 4 Patch: Solvability Check + Wider Dictionary

## Что изменилось

### 1. Solvability check
Генератор теперь **проверяет, что поле проходимо**: в каком-то порядке можно собрать минимум 7 спрятанных слов подряд (или все, если их меньше). Если поле не проходимо — генерируется заново.

### 2. Расширенный словарь
Словарь увеличен с 100 до **200 слов**. Это даёт генератору больше материала: на поле уверенно умещается 8–14 слов с пересечениями, а не 6–10 как раньше. Заметнее отличается каждый запуск.

---

## Как работает солвер

1. **Состояние доски** = битовая маска (`long`) клеток, ставших vacant. Стартовое — только центр.
2. **DFS**: на каждом шаге для каждого спрятанного слова проверяет можно ли его собрать сейчас:
   - Стартовая клетка имеет vacant-соседа
   - Ни одна клетка пути уже не vacant
3. Если можно — собираем (добавляем клетки в state) и идём глубже.
4. Memoization: посещённые состояния помечаются, чтобы не разбирать повторно.
5. Цель — достичь глубины 7 (или `placedWords.Count` если меньше).

### Стратегия retry
- **25 строгих попыток**: target = `min(placedWords.Count, 7)`. Если хотя бы одна цепочка такой длины существует — поле принимается.
- Если не получилось — **10 «расслабленных» попыток** с target = 3.
- Если и тут провал — берётся лучшая из попыток (самая длинная цепочка) с предупреждением в Console.

### Что солвер НЕ учитывает
- Альтернативные пути к слову через дубли букв (например, если на доске 2 буквы `T`, слово `CAT` можно собрать через любую из них). Используются только пути из генерации.
- Слова в шуме, не входящие в спрятанные. Игрок может найти их как бонус.

Это делает солвер **pessimistic** (может ошибочно отбраковать решаемое поле), но никогда не **optimistic** (если сказал «решаемо» — точно решаемо).

## Что увидишь в Console

При старте Game-сцены теперь несколько вариантов:

**Хороший случай (большинство):**
```
[BoardGenerator] Solvable (chain=7) after 1 attempts. 11 words: STARE, TRAIN, HEART, BOARD, RAIN, ...
```

**После нескольких retry:**
```
[BoardGenerator] Solvable (chain=7) after 4 attempts. 9 words: ...
```

**Очень редко (relaxed):**
```
[BoardGenerator] Solvable only at relaxed chain=3. 7 words: ...
```

**Совсем редкий случай (defensive fallback):**
```
[BoardGenerator] Could not reach target solvability. Returning best (chain=5, words=8).
```

## Расширенный словарь — 200 слов

Категории:
- 70 трёхбуквенных
- 90 четырёхбуквенных
- 40 пятибуквенных

Все слова — **без букв F, J, K, Q, X, Z** (эти буквы остались как шум — на пустых клетках). Шумовой набор сократился до 6 букв (было 8) — это компенсируется тем, что валидных слов в словаре теперь больше, и шумовых клеток на поле меньше (поле гуще заполнено осмысленными буквами).

Полный список можно посмотреть в `Assets/WordGame/Editor/SetupDictionaryIteration4.cs`.

## Установка

1. Распаковать архив поверх проекта.
2. Дождаться компиляции.
3. **Запустить editor-скрипт повторно: `WordGame > Setup Dictionary (Iteration 4)`**
   — теперь скрипт **перезаписывает** `word_list.txt` (раньше пропускал если файл уже есть). Это нужно, чтобы получить расширенный словарь.
4. Открыть `MainMenu.unity` → Play.

## Проверка качества

| Сценарий | Ожидание |
|---|---|
| Запустить игру 5 раз подряд | Каждый раз в Console `Solvable (chain=7)` или близко. Поле визуально насыщеннее, шума заметно меньше |
| Проверить лог с уложенными словами | 8–14 слов, разной длины |
| Попробовать собирать слова из лога | Всегда находится первый вариант рядом с центром, дальше можно продолжать цепочкой |
| Сравнить с предыдущей версией (если запомнил) | Меньше «мёртвых» полей где трудно стартовать |

## Параметры тюнинга (в `BoardGenerator.cs`)

- `MAX_ATTEMPTS_STRICT = 25` — попыток с строгим target
- `MAX_ATTEMPTS_RELAXED = 10` — попыток с relaxed target
- `TARGET_CHAIN = 7` — желаемая длина цепочки
- `RELAXED_CHAIN = 3` — fallback цель
- `MIN_WORDS = 5` — минимум слов на поле
- `NoiseLetters = { 'F', 'J', 'K', 'Q', 'X', 'Z' }` — шум

## Производительность

DFS с memoization занимает миллисекунды для типичного поля (10 слов, target=7). Не оптимизировал ничего экстра, и вряд ли понадобится — солвер вызывается один раз на запуск игры, и даже 25 retry укладываются в < 100мс на нормальной машине.

Если в будущем словарь вырастет до тысяч слов и/или поле станет больше (radius=4 или 5), может потребоваться оптимизация. Пока запас огромный.

---

## Что НЕ покрывает солвер

- **Escape mode (it.7):** там нужно достичь края поля. Текущая проверка только гарантирует возможность собрать N слов подряд. В it.7 добавлю отдельную проверку «достижим ли край поля через цепочку слов».
- **Number-клетки (it.5):** ограничение на минимальную длину слова на стартовой клетке. Когда добавлю — солвер тоже надо будет научить этому ограничению.

Эти доработки запланированы. Сейчас базовая гарантия проходимости работает.
